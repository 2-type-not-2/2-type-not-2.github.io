Time-Travel Experimentation on Static Type Systems for JavaScript


One smart sentence

Overview

Javascript has experienced explosive growth and is now used in large mature projects even outside the web domain. JavaScript is also a dynamically typed language for which static type systems, notably Microsoft’s TypeScript and FaceBook’s Flow, have been written. What benefits do these static type systems provide?

To answer this question, we employ a novel experimental methodology that augments mining software repositories with experimentation to quantify the proportion of bugs that these type systems can help avoid. Leveraging Javascript project histories, we “travel back in time” by selecting a fixed bug and checking out the code just prior to the fix. We then create an alternative history by adding type annotations to the buggy code and testing whether TypeScript and Flow would have detected the bug, presumably preventing its public release.

Methodology



Corpus Collection


We seek to construct a corpus of bugs that is representative and sufficiently large to support statistical inference. As always, achieving representativeness is the main difficulty and uniform sampling our main tool. We cannot sample bugs directly, but rather commits that we must classify into fixes and nonfixes. Why fixes? Because a fix is often labeled as such, its parent is almost certainly buggy and it identifies the region in the parent that a developer deemed relevant to the bug. To identify bug-fixing commits, we consider only projects that use issue trackers, then, in parallel, we look for bug report references in commit messages and commit ids (SHAs) in bug reports. This heuristic is not only noisy; it must also contend with bias in project selection and bias introduced by missing links.

Striving for generality, we used the standard sample size computation to determine the number of bugs needed to achieve a specified confidence interval. On 19/08/2015, there were 3,910,969 closed bug reports in JavaScript projects on GitHub. We use this number to approximate the population. We set the confidence level and confidence interval to be 95% and 5%, respectively. The result shows that a sample of 384 bugs is sufficient for the experiment, which we rounded to 400 for convenience.



Annotation


Procedure 1 defines our manual type annotation procedure. Because we annotate each bug twice, once for each type system, our experiment is a within-subject repeated measure experiment. As such, a phenomenon known as learning effects may come into play, as knowledge gained from creating the annotations for one type checker may aid in creating the annotation for the other. To mitigate the learning effects, for a bug b in B, we first pick a type checker tc from TypeScript and Flow uniformly at random, so that, on average, we consider as many bugs for the first time for each type system (line 3). If b is not type related “beyond a shadow of a doubt”, such as misunderstanding the specification, we label it as undetectable under tc and categorise it, skipping the annotation process. If not, we initialise the environment: for Flow, we add a header comment /* @flow */ to the file we want to annotate, whereas for TypeScript, we change the file extension from “.js" to “.ts” (line 4). We read the bug report and the patch to identify the patched region, a set of scopes are touched by the patch. Combining human program comprehension and JavaScript’s read–eval–print loop (REPL), e.g. Node.js, as appropriate, we conjecture a type annotation that enables tc to detect b. Following a set of predefined annotation tactics, we add the hypothesised annotations with the aim of driving tc to type check the region under assessment. We ignore type errors that we consider unrelated to this goal. Note that we are neither familiar with any project in our corpus, nor experts in type systems. To combat this, we have striven to be conservative: we annotate variables whose types are difficult to infer with any. Then we type check the resulting program. This process continues until we confirm that b’s bug is tc-detectable because tc throws an error within the patched region and the added annotations are nontrivial and consistent, or we deem b is not tc-detectable, or we exceed the time budget M.
